# Space и channel: единая сущность scope

Записка фиксирует решение о том, что канал — единственная сущность scope в протоколе, «сервер» — не отдельная сущность, а канал типа space, и адресация — плоские id.

## Отправная точка

Вопрос стоял так: что такое «сервер» в протоколе? В Discord и Slack «сервер» / «workspace» — группировка каналов с общим membership и governance (удалил участника из сервера → каскадный revoke из всех каналов). Нужен ли «сервер» как отдельная сущность протокола, или это понятие приложения?

В параллельной дискуссии по Corazon Protocol обсуждались два полюса: `server_id` как поле протокольного конверта (даёт governance «из коробки», но утолщает протокол) и server как чистый app-level (тонкий протокол, но каскадный revoke не гарантирован).

## Три варианта

**Вариант A: сервер — отдельная сущность протокола.** Транзакции `create_server`, `server_join`, `server_leave`. Нода знает двухуровневую иерархию: сервер → каналы.

Плюс: governance из коробки. Минус: протокол фиксирует ровно два уровня scope. Если нужен третий (организация → сервер → канал) — протокол не готов.

**Вариант B: сервера нет, только каналы.** Протокол знает `channel` и `membership`. «Сервер» — convention приложения. 20 каналов для протокола — 20 независимых сущностей.

Плюс: максимально тонкий протокол. Минус: каскадный revoke не гарантирован. Плохо написанный клиент может «забыть» удалить из части каналов.

**Вариант C: сервер — это канал.** Канал типа `space`, чей membership определяет, кто может быть в дочерних каналах. Дочерние каналы ссылаются на parent через `parent_scope`. Нода при валидации `join` в дочерний канал проверяет: pubkey состоит в parent_scope? Если нет — отклоняет.

Плюс: не вводит новую сущность. Использует ту же механику (транзакции, membership, parent-ссылка). Рекурсивно: третий уровень — ещё один space, parent для «серверов», без изменения протокола. Минус: семантическая перегрузка — канал, в котором общаются, и канал-контейнер выглядят одинаково.

## Принятое решение: вариант C

Канал — единственная сущность scope. Два типа:

**`conversation`** — канал, в котором общаются. Содержит сообщения (structure).

**`space`** — канал-контейнер. Содержит другие каналы (дочерние по `parent_scope`). Его membership определяет потолок: в дочерних каналах могут состоять только те, кто состоит в parent space.

Тип — атрибут канала (`channel_type`), не отдельная сущность. Различие решает семантическую перегрузку без введения нового уровня в протоколе.

## Один паттерн, два уровня

Сообщения — одна сущность (`message`), дерево через `reply_to`. Каналы — одна сущность (`channel`), дерево через `parent_scope`. Тот же принцип: минимум сущностей, максимум выразительности через связи. Решение из сессии `2026-04-08-message-bus` (единый message, дерево через parent) здесь применено на уровне каналов.

## Каскадный governance

Каскадный revoke — вычислимое следствие модели. Если pubkey удалён из space, нода пересчитывает membership всех дочерних каналов. Это валидация протокола, не логика приложения.

Каскадный `join` работает в обратном направлении: чтобы вступить в дочерний канал, нужно сначала состоять в parent space. Нода проверяет это при обработке `join`.

## Терминология: scope и space

В рамке scope/structure/discipline слово «scope» означает **измерение** — способ смотреть на канал через призму membership. Слово «space» означает **тип канала** — канал-контейнер. Это разные вещи: scope есть у любого канала (и у conversation, и у space), а space — конкретный тип.

## Адресация

Адрес сообщения в протоколе: **`channel-id/msg-id`**. Два плоских идентификатора. Протокол не знает и не кодирует в адресе, в каком space живёт канал.

Адрес канала: **`channel-id`**. Канал типа conversation и канал типа space адресуются одинаково. Тип — атрибут, не часть адреса.

Приложение строит навигацию из дерева scope: «Мой space > #general > сообщение от Alex». Это display path, не адрес. Переименовал space — display path поменялся, адрес нет. Перенёс канал в другой space — display path поменялся, адрес нет.

Принцип один на обоих уровнях: адрес сообщения не включает предков в дереве structure (решение из сессии `2026-04-08-message-bus`), адрес канала не включает предков в дереве scope. Контекст — вопрос навигации, не идентификации.

## Терминология протокола и терминология UX

Протокол знает: channel, space, conversation, parent_scope, channel-id. Это инструменты.

Приложение может называть space «сервером» (Discord), «workspace» (Slack), «пространством», «комнатой», чем угодно. Может показывать рекурсивность (вложенные space) или скрывать её (верхний space = «сервер», вложенные = «категории»). Протокол не диктует UX.

Это сила, а не компромисс: один конструктор, сколько угодно продуктов из него.
