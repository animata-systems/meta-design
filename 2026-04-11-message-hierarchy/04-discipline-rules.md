# Правила discipline: конкретный набор и наследование

Записка фиксирует конкретный набор правил discipline, их проверку на сценариях и модель наследования через дерево scope.

## Набор правил

Пять правил. Каждое — одно поле, одна проверка на ноде. Набор проверен на четырёх сценариях (рабочий space, публичное сообщество, игровой сервер, приватный чат). Принцип: расширить набор всегда легче, чем сузить. Начинаем с минимума.

**`max_depth`** (число). Максимальная глубина дерева сообщений в канале. Нода при получении `message` проверяет: глубина по цепочке `reply_to` ≤ max_depth. Отклоняет, если превышает.

Покрывает: рабочий space (5–10), приватный чат (1 = линейный поток), сообщество (большое число или без лимита).

**`branching`** (on / off). Можно ли отвечать не на последнее сообщение в ветке. Off = строго линейный поток: каждое следующее сообщение может быть только ответом на предыдущее. On = свободное ветвление.

Покрывает: игровой лог (off), приватный чат (off), рабочее обсуждение (on).

**`post_permission`** (any_member / authority_only). Кто может отправлять сообщения в канал. authority_only = только участники с authority.

Покрывает: канал объявлений (authority_only), всё остальное (any_member).

**`thread_creation`** (any_member / authority_only). Кто может создавать корневые сообщения (треды). Отдельно от post_permission: в модерируемом канале все могут отвечать внутри тредов, но новые треды создаёт только authority.

Покрывает: модерируемое сообщество (authority_only создаёт темы, участники обсуждают), канал задач (authority создаёт задачу-тред, участники работают внутри).

**`reaction_permission`** (any_member / authority_only / off). Можно ли реагировать на сообщения. Отдельно, потому что реакции — входные данные для SymbioRank. Off = реакции отключены.

Покрывает: игровой лог (off — реакции бессмысленны), канал объявлений (any_member — реакции как фидбек при закрытом post_permission).

## Проверка на сценариях

**Рабочий space.** `max_depth: 10, branching: on, post_permission: any_member, thread_creation: any_member, reaction_permission: any_member`. Свободное обсуждение. Канал объявлений внутри того же space: `post_permission: authority_only, reaction_permission: any_member`.

**Публичное сообщество.** Основной канал: `thread_creation: any_member, branching: on, max_depth: 5`. Модерируемый канал: `thread_creation: authority_only, post_permission: any_member` — authority создаёт темы, участники обсуждают.

**Игровой сервер.** Лог событий: `branching: off, max_depth: 1, post_permission: authority_only, reaction_permission: off`. Чат игроков: `branching: on, max_depth: 3, post_permission: any_member`.

**Приватный чат.** `max_depth: 1, branching: off, post_permission: any_member, thread_creation: any_member, reaction_permission: any_member`. Линейный поток между двумя участниками.

Все четыре сценария покрыты пятью правилами. Шестое правило не понадобилось.

## Наследование discipline: копирование при создании + ceiling

**Копирование при создании.** Когда создаётся дочерний канал, он копирует текущую discipline из parent space как дефолт. Дальше живёт независимо. Если parent space меняет свои правила — на существующие дочерние каналы это не влияет, только на вновь создаваемые. Согласуется с принципом «закон обратной силы не имеет».

**Ceiling.** Parent space может задавать потолок — максимально допустимые значения правил для дочерних каналов. Дочерний канал может быть строже потолка, но не мягче. Нода при валидации `set_rule` в дочернем канале проверяет: новое значение ≤ ceiling parent space.

Ceiling работает однонаправленно, как и discipline относительно scope: сужает, но не расширяет. Parent space задаёт рамку, дочерний канал работает внутри неё.

**Ужесточение ceiling.** Если parent space хочет ужесточить ceiling, а какой-то дочерний канал уже шире — транзакция `set_ceiling` отклоняется. Сначала нужно сузить дочерние каналы, потом сужать ceiling. Тот же принцип, что и при изменении discipline: привести состояние к совместимому — задача приложения.

**Проверка ceiling — одна проверка вверх.** Нода при валидации `set_rule` поднимается на один уровень (к parent space) и проверяет ceiling. Не обход всего дерева — только один шаг. Приемлемая стоимость за реальный governance.

## Допущения

Набор из пяти правил — черновой. Проверен на четырёх сценариях, но реальное использование может выявить пробелы. Принцип «расширить легче, чем сузить» позволяет начать с минимума и добавлять по мере необходимости.

Ceiling принят с оговоркой: модель ощущается правильной, но edge cases при каскадном ужесточении (space → дочерний space → каналы) требуют проработки.
