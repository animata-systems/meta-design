# 03. Инструменты как первоклассный слой

Это, возможно, самый сильный сдвиг в понимании системы за всю сессию. В ранней картине инструменты выглядели как «деталь языка COIL — операция ВЫПОЛНИ, которой пользуется протокол». На самом деле это **отдельный слой архитектуры**, который используется не только протоколами.

## Центральная мысль

> **COIL-приложение — это интерфейс к набору инструментов, у которого есть два способа взаимодействия: человеческий (через UI) и интеллектуальный (через протоколы и сообщения).**

Это определение снимает огромную часть путаницы. Приложение — не контейнер с данными, не сервер, не процесс. Это **пара входов** (UI и протокол), которые оба дотягиваются до одного и того же внешнего мира через инструменты.

## Симметрия фронтенда и протокола

Один и тот же инструмент может быть вызван как из фронтенда, так и из протокола. Фронтенд нажимает кнопку — вызывает `создать_приложение`. Протокол в ходе размышления — вызывает тот же `создать_приложение`. Результат один, потому что инструмент — это единая точка правды.

Это значит, что у приложения **нет «внутреннего состояния»**, которое надо синхронизировать между фронтендом и протоколом. Состояние живёт за инструментом. Оба клиента (фронтенд и протокол) читают и пишут в одно и то же место. Нет гонок, нет двойной записи, нет «какой из двух источников правды актуальнее».

## Состояние живёт за инструментами

В традиционной архитектуре состояние приложения живёт либо на клиенте, либо на сервере, либо в БД. У нас — ни в одном из этих мест. Состояние — это то, что **возвращает инструмент** при чтении и то, что **меняет инструмент** при записи. Инструмент — единственный владелец.

Из этого вытекает несколько свойств:

- Приложение не держит процесс. Оно существует как запись в реестре и оживает, когда его кто-то открывает или когда в его канал приходит сообщение.
- Состояние переживает перезапуск фронтенда, перезапуск хоста Corazon, перезапуск протокола. Оно не в них.
- Два экземпляра одного приложения, открытые в разных браузерах, видят одно и то же состояние — потому что оба смотрят в один и тот же инструмент.
- Протокол может «догнать» текущую ситуацию, просто прочитав инструмент. Не нужен реплей, не нужна синхронизация.

## Два канала взаимодействия — одна модель мира

Сообщения и инструменты играют разные, дополняющие друг друга роли:

| Сообщения (Corazon) | Инструменты |
|---|---|
| **Что произошло** | **Как сейчас обстоит дело** |
| События, факты, неотменимые | Состояние, меняемое, запрашиваемое |
| Координация | Владение данными |
| Подписи, порядок, неопровержимость | Актуальность, действие |
| Автор события зафиксирован | Автор изменения может быть проверен через лог вызовов |
| Детерминированно наблюдаемы | Могут быть недетерминированными (внешний мир) |

Приложение использует **оба** канала. Если что-то надо скоординировать между несколькими участниками — это сообщение в канале. Если что-то надо сделать или прочитать во внешнем мире — это вызов инструмента.

## Почему «дизайнер приложений» сработал

Первое приложение, написанное руками, — это «дизайнер приложений». У него есть фронтенд (куда пользователь пишет, какое приложение ему нужно), набор инструментов (`создать_приложение`, `сохрани_код`) и протокол `coder.coil`, который умеет думать план, думать веб, думать протокол и вызывать инструменты.

Ключевая вещь: в этой конструкции **нет специальной инфраструктуры создания приложений**. Есть просто инструмент, который умеет записать новое приложение в реестр. Фронтенд и протокол пользуются этим инструментом одинаково. Поэтому первое приложение смогло создать все остальные — не потому что мы построили мета-уровень, а потому что **не строили его**. Создание приложения — это такое же действие, как размещение заказа или проведение платежа.

## Четыре текущих приложения — одна структура, разные пропорции

| Приложение | Центр тяжести | Использует сообщения? |
|---|---|---|
| Дизайнер приложений | Протокол + инструменты | Да (канал для ДУМАЙ) |
| Desktop Cora OS | Только инструменты (`мои_приложения`) | Нет |
| App Store | Только инструменты (`доступные_приложения`, `добавить_мне_приложение`) | Нет |
| Оракул | Протокол + канал | Да |

Важное наблюдение: **приложение не обязано использовать сообщения вообще**. Если приложение — это чистое окно в инструмент (как Desktop или App Store), оно может жить без единого канала и без единого протокола. Каналы и сообщения появляются тогда, когда нужна интеллектуальная реакция или координация между участниками.

Это важно для ментальной модели разработчика приложений: **не каждое приложение — это чат с агентом**. Многие приложения могут быть просто удобными UI над инструментами.

## Три возможных будущих для слоя инструментов

Сессия закончилась на том, что пользователь описал экономическую модель, в которой **доступ к инструменту можно продавать** (см. `04-three-layer-economy.md`). Но до этого был задан открытый вопрос: откуда берутся инструменты в будущем, когда их перестанут писать руками.

Три сценария, все три могут сосуществовать:

1. **Инструменты как часть приложения.** Разработчик генерирует MCP-сервер с инструментами вместе с фронтендом и протоколами. Инструменты живут с приложением, видны только ему.
2. **Инструменты как отдельная экосистема.** Есть публичный каталог (что-то вроде npm для MCP). Приложение собирается как композиция из UI + протоколов + ссылок на сторонние инструменты.
3. **Инструменты как часть операционной системы.** Cora OS поставляет базовый набор (реестр, файлы, identity, уведомления), а приложения строятся поверх них с редкими собственными добавлениями.

После разговора про экономику стало ясно, что **второй сценарий — не опциональный**. Как только появляется возможность продавать доступ к инструменту, немедленно появляется рынок инструментов как независимый слой. Вопрос не «будет ли он» — вопрос «насколько он будет централизован или распределён, и какое место в нём займёт Cora OS».

## Следствия для архитектуры, которые теперь обязательны

- Инструменты нужно обсуждать отдельно от протоколов, с собственной моделью идентичности, прав, владения и доступа.
- У каждого инструмента должна быть верифицируемая идентичность — агент должен быть уверен, что он говорит с тем самым инструментом, за который заплатил.
- Должна быть предусмотрена модель разрешений: «агент X имеет право вызвать инструмент Y на условиях Z», с возможностью отзыва.
- Cora OS, поставляя системные инструменты, становится первым примером провайдера инструментов. Это нужно осознанно проектировать, потому что системные инструменты задают шаблон для всех будущих.
