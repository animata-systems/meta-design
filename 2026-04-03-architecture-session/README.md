# Архитектурная сессия — 2026-04-03

Запись обсуждения распределённой архитектуры системы: из каких слоёв она состоит, как устроена легитимность, кто что подписывает, кто что реплицирует, и какие компромиссы принимаются в первой версии. Не лог, а реконструкция — разложена по темам.

Это сессия, в которой впервые был собран цельный взгляд на систему как на набор **проекций**: каждая проекция показывает один архитектурный слой, не пытаясь объяснить всё сразу. Этот каталог восстанавливает **рассуждения, которые за проекциями стоят**.

## Карта документов

| Файл | О чём |
|---|---|
| `01-layers.md` | Пять слоёв системы: идентичность/authority, server control, header log, payload store, sandbox. Почему они отделены друг от друга. |
| `02-governance-and-authority.md` | Root Authority, Delegated Authority, одношаговая нетранзитивная делегация, принцип «root управляет будущей легитимностью, а не прошлыми фактами». |
| `03-identity-and-participation.md` | Три типа идентичности: participant, node, app. Почему «пишет participant, а доставляет node» — это не мелочь. |
| `04-node-roles.md` | Четыре роли узлов: client/listener, relay, steward, authority. Почему участие в сети ≠ governance permission, а репликация ≠ authority. |
| `05-message-model.md` | Двухслойная модель сообщения: header event в Corazon + payload blob в хранилище. Форма header event, адресация, связи. |
| `06-execution-semantics.md` | Mention как триггер протокол-инстанса. Read path и write path агента. Почему mention — это не wake-up, а spawn. |
| `07-revocation-and-epochs.md` | Epoch-ключи на уровне канала. Что именно отзывается при revoke и что остаётся читаемым. Принцип асимметрии прошлого и будущего. |
| `08-brand-architecture.md` | Трёхслойный бренд: COIL (язык) → Corazon Network (субстрат) → Cora OS (операционное окружение). Зачем разделять и где граница. |
| `09-v1-compromises.md` | Что в v1 сознательно упрощено: метаданные, членство, payload store, revoke, удаление, канонический порядок. Где проходит линия «не MVP, но не полный идеал». |
| `open-questions.md` | Открытые вопросы, оставшиеся после сессии. |

## Как читать

Если сначала хочется понять **из каких слоёв состоит система** — `01` → `05` → `06`.

Если хочется понять **как устроена легитимность** — `02` → `03` → `04`.

Если хочется понять **какие компромиссы приняты сейчас** — `07` → `09`.

Если хочется понять **почему три имени (COIL/Corazon/Cora OS), а не одно** — `08`.

Это **рабочий снимок понимания на 2026-04-03**, не нормативный документ. Всё, что описано здесь, — контекст и обоснования. Когда какое-то решение дозреет до нормы — оно должно быть вынесено в соответствующий слой документации (`coil/spec/` или `coil-runtime/DESIGN.md`).
