# Консолидация идей из трёх предыдущих сессий

Задача файла — собрать идеи из сессий 03.04 (architecture), 07.04 (system model) и 08.04 (message bus), удалить дубликаты, слить близкие формулировки в одну и разметить оставшиеся конфликты. Это не новая архитектура, это **наведение порядка в уже сделанном мышлении**.

## Метод

Каждую идею беру из её источника и маркирую:

- **Уникальная** — появляется только в одной сессии, ни с чем не конфликтует.
- **Подтверждённая** — повторяется в двух или трёх сессиях, формулировки сливаются.
- **Смещённая** — тот же тезис, но одна из сессий уточнила формулировку; берём позднюю.
- **Конфликт** — сессии расходятся, требуется решение.

Источники сокращённо: `03` = `2026-04-03-architecture-session`, `07` = `2026-04-07-architecture-session`, `08` = `2026-04-08-message-bus`.

## Группа A — слоистость системы

**A1. Система разделена на слои, каждый со своим жизненным циклом.** (подтверждённая: `03/01`, `07/01`)

`03` предлагает пять слоёв: identity/authority, server control, header log, payload store, sandbox. `07` предлагает четыре компонента: Cora OS, Corazon Network, Sandbox, Blob storage. Разногласия нет — `07` смотрит сверху (компоненты), `03` — изнутри (слои внутри Corazon). Финальная единая схема:

```
Cora OS              (edge для людей, identity, gateway, фронтенды приложений)
Corazon Network      (A. Authority log + B. Server control log + C. Header log)
Blob storage         (D. Payload)
Sandbox              (E. Execution)
```

Линия границы между Cora OS и Corazon Network — ключевая: Cora OS — **не внутри** Corazon Network, а **построена над ней**. Слои A–E — внутренняя структура Corazon + storage + sandbox.

**A2. Cora OS — это iOS для COIL-приложений.** (уникальная: `07/01`)

Не сервер приложения, не контейнер, не runtime агентов. Операционная система для приложений: identity людей, хост фронтендов, gateway транзакций, системные инструменты. Оставляем в сильной формулировке `07`.

**A3. Sandbox — независимый runtime агентов, а не часть Cora OS.** (подтверждённая: `03/06`, `07/01`)

Может быть много sandbox-ов в системе, независимых, с разными ключами и разными операторами. Агенты живут у своего оператора и **приходят** в приложения как участники. Критически важная инверсия относительно обычной SaaS модели.

**A4. Blob storage — отдельный слой, не часть Cora OS.** (подтверждённая: `03/01`, `07/01`)

Шифрование на endpoint, ключи никогда не попадают в Cora OS, в Corazon лежит только идентификатор blob. Следствие: «может ли Cora OS читать сообщения» — решено не криптографией, а **топологически**.

## Группа B — идентичность и authority

**B1. Root Authority ↔ Delegated Authority, одношаговая нетранзитивная делегация.** (уникальная: `03/02`)

Root управляет **будущей** легитимностью, не переписывает прошлые факты. Authority делегирована одношагово — нет цепочки «authority делегирует authority делегирует authority».

**B2. Три типа идентичности: participant, node, app.** (уникальная: `03/03`)

«Пишет participant, а доставляет node» — граница, которая делает возможной p2p-репликацию без того, чтобы узел получал authority на содержимое.

**B3. Четыре роли узлов: client/listener, relay, steward, authority.** (уникальная: `03/04`)

Участие в сети ≠ governance permission. Репликация ≠ authority. Разделение ролей — инструмент против смешения «я вижу → я решаю».

**B4. Шесть ролей акторов системы.** (уникальная: `07/02`)

Разработчик протокола, разработчик приложения, оператор агента, участник, пользователь, оператор узла Corazon. Это **социальный** взгляд, дополняющий **технический** `03/03`. Не конфликт, а разные оси.

**B5. Верифицируемая идентичность на всех трёх слоях экономики.** (уникальная: `07/04`)

Инструмент, агент, приложение — каждый верифицируем. Это условие существования рынка, не опциональная фича. Сливается с **B1** в требование единой identity-модели, покрывающей и governance, и экономику.

## Группа C — сообщения и разговоры

**C1. Двухслойная модель сообщения: header в Corazon + payload в blob store.** (уникальная: `03/05`)

Приватность отделена от маршрутизации: факт сообщения распространяется, содержимое — нет.

**C2. Унифицированная сущность `message` с опциональным `parent`; `thread` — сообщение без родителя.** (уникальная: `08/04`)

Закрывает вопрос плоской vs иерархической модели. Дерево реплик — не отдельная сущность, а побочный эффект `parent`. Это поздняя формулировка, она заменяет любые ранние намёки на `post`/`comment`/`reply` как разные типы.

**C3. Симметризм: агент и человек — равные участники разговора.** (уникальная, но центральная: `08/03`)

Один тип сущности — `message`. Один механизм `@mention`. Оба могут упоминать обоих. Внутренняя жизнь у обоих приватна по умолчанию. Все документы `08` выводятся из этого хода.

**C4. Приватное размышление агента в sandbox не публикуется по умолчанию.** (уникальная: `08/06`)

Эквивалент внутреннего монолога человека. Но с правилом: **если агент добровольно публикует трейл, среда не должна мешать**. Прозрачность — атрибут участника, не обязательство среды.

**C5. Mention как триггер протокол-инстанса (spawn, не wake-up).** (уникальная: `03/06`)

Каждое упоминание — новый инстанс, не «разбудили старую сессию». Это развивается в `05-agent-context.md` сессии `08`: агент видит цепочку предков до корня треда.

## Группа D — инструменты

**D1. Инструменты — первоклассный слой, не деталь языка.** (уникальная: `07/03`)

Сдвиг из `07`: инструмент — не операция «ВЫПОЛНИ» в COIL, а отдельная архитектурная абстракция. Состояние живёт **за** инструментами, не в приложении.

**D2. Симметрия фронтенда и протокола.** (уникальная: `07/03`)

Один и тот же инструмент вызывается из UI человека и из протокола агента. Это исключает классическую двойную запись «UI state vs backend state».

**D3. Три сценария происхождения инструментов.** (уникальная: `07/03`)

Внутри приложения, из публичного каталога (независимая экосистема), из Cora OS (системные). Все три сосуществуют, но **второй обязателен** при появлении экономики.

## Группа E — экономика

**E1. Трёхслойный рынок: tools → agents → apps.** (уникальная: `07/04`)

Деньги сверху вниз, ценность снизу вверх. Никто не знает чужих слоёв. Это **главный** экономический вывод из `07`.

**E2. Экономический слот в сообщении: опциональное расширение baseline.** (уникальная: `08/07`)

Обязательство / запрос / квитанция / эскроу. Pluggable backends (x402, ACP, AP2, MPP, банки). Corazon не владеет расчётами, но обеспечивает атомарность доставки и обязательства. Это **механизм**, через который реализуется **E1**.

**E3. Мотивация узлов Corazon — комиссия с того, что проходит, не собственная валюта.** (уникальная: `08/07`)

Нейтральность протокола сохраняется; комиссия — свойство оператор-ского соглашения, не протокольного поведения.

**Слияние E1 + E2 + E3:** трёхслойный рынок — экономическая модель, экономический слот — протокольный механизм её выражения, комиссия узлов — источник мотивации операторов сети. Три уровня описания одного и того же, не конфликт.

## Группа F — бренд и рамка

**F1. Трёхслойный бренд: COIL (язык) → Corazon Network (субстрат) → Cora OS (операционное окружение).** (подтверждённая: `03/08`, `08/01`)

Без разногласий. `08` усиливает: три слоя — это три разных **по природе** вещи, не три продукта одной компании.

**F2. Нейтральная инфраструктура координации, не вертикальный стек.** (уникальная, но рамочная: `08/01`)

Позиция «TCP/IP для агентов», не «Stripe для агентов». Принимается как ограничение, применяемое ко всем остальным решениям.

**F3. COIL enterprise-ready как язык — раньше, чем сеть массовая.** (уникальная: `03/08`)

«COIL не делает агентов умнее — COIL делает их управляемыми». Это **продажа** языка отдельно от продажи сети. Важный стратегический ход, который сессии `07` и `08` не отменяют, но и не развивают.

## Группа G — компромиссы первой версии

**G1. Метаданные header-событий видимы всем, кто реплицирует канал.** (`03/09`)

**G2. Полудецентрализованное членство — authority управляет списком участников.** (`03/09`)

**G3. Payload storage централизован через steward-ов (Animata + партнёры).** (`03/09`)

**G4. Revoke нелокальный, не мгновенный.** (`03/09`)

**G5. Soft delete, не hard delete.** (`03/09`)

**G6. Sequencer для канонического порядка; «почти-цепочка» вместо DAG.** (`03/09`)

Все шесть компромиссов выбраны по одному критерию: «можно ли уйти к идеалу позже, не ломая контракт?» Это **осознанный долг**, не недосмотр. Группа не конфликтует ни с чем из последующих сессий.

## Конфликты и напряжения

Настоящих противоречий между сессиями нет. Есть три напряжения, которые нужно явно проговорить:

**1. `07` продвигает «инструменты первого класса», `08` этим почти не пользуется.**
`08` про message bus, у него свой фокус. Но нужно проверить, что симметризм и экономический слот не опираются **только** на сообщения. Пример: если инструмент тоже может нести экономический слот (вызов `инструмент → деньги`), это должно быть в модели. Сейчас **не проговорено**, отнесено в открытые вопросы.

**2. `03` говорит «sandbox слушает header-лог и спавнит COIL-инстансы», `08` говорит «агент размышляет в sandbox, наружу выходит только сообщение».**
Не конфликт, но важно зафиксировать: trigger — header event, cognition — внутри sandbox, output — снова header event. Sandbox — **мостик** между внешним наблюдаемым и внутренним приватным.

**3. `03/09` фиксирует «метаданные видимы всем», `08/06` вводит правило приватности трейла.**
Разные уровни: `03` про метаданные header-сообщений (кто, кому, когда), `08` про приватность размышления агента. Оба утверждения истинны одновременно: метаданные публичны, размышление приватно. Но нужно **один раз** это явно проговорить в спеке, иначе будет путаница.

## Сухой список того, что консолидировалось

После слияния и дедупликации список базовых идей системы выглядит так. Это короткий индекс, пригодный для проверки любого нового решения «помещается оно в систему или нет».

1. **Три имени — три аудитории.** COIL / Corazon / Cora OS.
2. **Нейтральная инфраструктура координации.** Не вертикальный стек.
3. **Симметризм участников.** Агент и человек — равные в разговоре.
4. **Message — единая сущность** с опциональным `parent`.
5. **Приватность размышления.** Трейл агента в sandbox не публикуется по умолчанию.
6. **Слои Corazon отделены от payload и sandbox.** Пять слоёв внутри + payload + sandbox снаружи.
7. **Инструменты — первоклассный слой.** Состояние живёт за ними.
8. **Симметрия фронтенда и протокола** в доступе к инструментам.
9. **Трёхслойный рынок:** tools → agents → apps.
10. **Экономический слот — опциональное расширение**, не baseline.
11. **Pluggable payment backends.** Corazon не владеет расчётом.
12. **Root authority + одношаговая делегация.**
13. **Три типа идентичности:** participant, node, app.
14. **Шесть ролей акторов.** Социальная проекция identity.
15. **Mention — spawn**, не wake-up.
16. **Компромиссы v1 — осознанный долг**, не недосмотр.
17. **COIL как продукт для enterprise** идёт отдельно от сети.

Эти 17 пунктов — **содержание** того, что мы решили за последние две недели. Всё остальное, что будет обсуждаться, должно либо укладываться сюда, либо явно что-то из этого двигать.
