# Переосмысление архитектуры: открытая сеть, хосты и платформа

Эта записка фиксирует архитектурный сдвиг, который произошёл при проектировании Corazon Protocol как самостоятельной открытой сети. Предыдущие сессии описывали систему как трёхслойный стек (COIL → Corazon → Cora OS). Здесь рамка меняется: Corazon Network — самостоятельный субстрат, а всё остальное — клиенты этой сети.

## Главный сдвиг: Corazon Network как самостоятельная открытая сеть

Corazon Network не предъявляет требований к тому, на чём написан агент. Сеть оперирует транзакциями: подписанными заголовками с метаданными (кто, кому, когда, тип, ссылка на payload). Чтобы стать участником сети, достаточно уметь три вещи: управлять identity-ключами, подписывать транзакции, получать и обрабатывать header-события.

COIL как язык в этой картине — не обязательное условие входа в сеть, а **слой доверия**. Агент, написанный на COIL, верифицируем: его протокол можно прочитать, его бюджет предсказуем, его поведение аудитабельно. Агент на любом другом рантайме тоже может участвовать в сети — но он чёрный ящик. Выбор за оператором: хочешь доверие — пиши на COIL. Не хочешь — входи как есть, но рейтинг это отразит.

Это принципиально отличается от модели «COIL — язык сети». COIL остаётся ценным, но его ценность — не в монополии на вход, а в том, что он даёт участнику. Продажа через ценность, а не через замок.

## Хост как кошелёк

В предыдущих сессиях Corazon Host описывался как «независимый runtime агентов». Теперь понимание обобщается: **хост — это интерфейс к сети**, аналогичный кошельку в криптомире.

Кошелёк в крипто управляет ключами, подписывает транзакции, отображает балансы, но сам не является блокчейном. Хост в Corazon управляет identity-ключами, подписывает транзакции, отображает рейтинг и каналы, но сам не является сетью. Хостов может быть много, разных, от разных разработчиков, для разных задач:

- Хост для COIL-агентов — оптимизирован под COIL-runtime, budget enforcement, governance.
- Плагин для существующего агентского фреймворка (OpenClaw, LangGraph, CrewAI) — минимальный клиент, который позволяет агентам этого фреймворка участвовать в сети Corazon.
- Cora OS — flagship-хост для людей (подробнее ниже).
- Любой кастомный хост, реализующий протокол.

Все они равноправны на уровне сети. Нода Corazon не знает и не заботится, какой хост отправил транзакцию — она видит только подписанный заголовок.

## Cora OS: одновременно кошелёк и платформа

Cora OS — не просто один из хостов. Снаружи (по отношению к сети) — это хост, который подписывает транзакции от имени людей. Внутри (по отношению к пользователю) — это полноценная платформа с приложениями, магазином приложений и дизайнером приложений.

Аналогия с iOS здесь точна: App Store для публикации и обнаружения приложений, Designer для создания новых приложений прямо внутри среды, приложения как единица пользовательского опыта. Но это iOS, который одновременно является кошельком — управляет identity, подключает к открытой сети, подписывает транзакции.

Ни один из текущих кошельков не даёт создавать приложения внутри себя. Ни одна из текущих мобильных платформ не подключена к открытой сети транзакций, где агенты и люди — равные участники. Cora OS — и то, и другое: **платформа для нового мира, где создание приложения — операция внутри приложения**.

Designer внутри Cora OS — это приложение, в котором человек и агент-помощник вместе собирают новое приложение. Это прямое следствие симметризма: если агент и человек — равные участники, то и в процессе создания они равные. Созданное приложение публикуется в App Store и становится доступно другим участникам.

## Стратегическая асимметрия

Войти в Corazon Network может кто угодно через минимальный хост. Но полный опыт — создание приложений, магазин, симбиоз с агентами-помощниками, визуальный Designer — есть только в Cora OS.

Это создаёт двухуровневую стратегию роста:

Первый уровень — **сетевой эффект через открытость**. Каждый плагин для существующего фреймворка приносит в Corazon всех пользователей этого фреймворка. Не нужно убеждать их переписать агентов на COIL — нужно убедить подключиться к сети. Порог входа минимален.

Второй уровень — **глубина через Cora OS**. Когда участник сети хочет больше, чем просто отправлять и получать транзакции — хочет создавать приложения, использовать магазин, получить полный пользовательский опыт — он приходит в Cora OS. Это как разница между любым email-клиентом и Gmail: протокол один, но платформа даёт экосистему.

## Что изменилось по сравнению с предыдущими сессиями

Сессия `2026-04-09` зафиксировала 17 консолидированных пунктов. Вот что из них сдвигается:

**A2 (Cora OS — iOS для COIL-приложений)** — уточняется. Cora OS остаётся платформой с приложениями (iOS-аналогия верна), но одновременно является хостом-кошельком, подключённым к Corazon Network. Не «iOS для COIL-приложений», а «iOS + кошелёк для участников Corazon». Приложения внутри Cora OS не обязаны быть написаны на COIL — COIL опционален, но повышает доверие.

**A3 (Corazon Host — независимый runtime агентов)** — обобщается. «Host» становится generic-понятием: любой клиент, реализующий протокол. Corazon Host для COIL-агентов — частный случай хоста, оптимизированный под COIL-runtime. Cora OS — другой частный случай, оптимизированный под людей. Плагин для OpenClaw — третий.

**F1 (трёхслойный бренд COIL / Corazon / Cora OS)** — меняет природу. Не три слоя стека, а три вещи разной природы: язык доверия (COIL), открытая сеть (Corazon Network), flagship-платформа для людей (Cora OS). Они связаны, но не вложены друг в друга иерархически.

**R6 (декомпозиция Cora OS)** — снимается как проблема. Если Cora OS — хост + платформа (а не операционная система, несущая identity + authority + gateway + hosting + tools + root authority), её внутренняя сложность — это сложность хорошего кошелька с экосистемой приложений, а не сложность OS. Масштаб задачи существенно меняется.

Всё остальное из 17 пунктов остаётся без изменений: симметризм (C3), нейтральность (F2), экономический слот (E2), компромиссы v1 (G1–G6), приоритет одиночного оператора — всё работает и усиливается в новой рамке.

## Допущения

Эта записка держится на двух допущениях:

1. **Протокол Corazon достаточно прост, чтобы плагин для стороннего фреймворка можно было написать за разумное время.** Если минимальный клиент требует реализации сотен страниц спецификации — открытость фиктивна. Нужно будет проверить при проектировании wire protocol.

2. **Сетевой эффект от открытости перевешивает потерю контроля над качеством участников.** Открытая сеть означает, что в неё придут спамеры, плохие агенты, мусор. SymbioRank (см. `02-symbiorank.md`) — предполагаемый ответ на это, но его эффективность пока гипотеза.
